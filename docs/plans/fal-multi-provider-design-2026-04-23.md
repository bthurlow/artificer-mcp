## Context

Artificer-mcp is a shipped open-source MCP server that wraps generative media tools for AI callers. Today it covers Google's stack (Veo video, Gemini/Nano Banana image, Lyria music, Gemini TTS) plus ImageMagick and ffmpeg post-processing. The immediate motivating need is fal.ai support for audio-driven lip-sync models that solve a silence problem in the content pipeline. But the deeper need surfaced during design review: **as media providers multiply (fal, Google, ElevenLabs, Stability, Anthropic-hosted, etc.) and each provider exposes multiple models with distinct prompt languages, callers need a unified discovery layer that preserves artificer's agnostic-transport principle.**

This design expands scope from "fal adapter" to "multi-provider architecture" because the architectural choices that get made during fal integration ossify the project's trajectory. Better to commit to the right abstraction once than to retrofit.

## Problem Statement

Two coupled problems:

**Immediate (talking-head):** The content pipeline's talking-head generator is stuck on a Veo 3.1 behavior — the model synthesizes audio along with video, ignores silence directives, caps downstream audio-split crossfade at 0.25s. Fixable only by using a video model that accepts audio as *input* (TTS URL → lip-synced video out) instead of synthesizing audio alongside video. Fal.ai hosts such models.

**Structural:** Artificer's current tool surface is provider-scoped (`gemini_generate_video`, `gemini_generate_image`, etc.) with no discovery layer. Adding fal creates inconsistency unless we simultaneously commit to the cross-provider pattern: models are logical entities with their own prompt languages, providers are access routes, the same model can be reachable via multiple providers (e.g., Veo via Google's API *or* via fal), and callers need to discover what's actually available on their configured keys.

## What Makes This Cool

- **Decoupling audio from video** (the talking-head unlock). Pipeline authors the TTS track with any cadence and silence it wants; the model animates to it verbatim. Silence problem disappears by construction.
- **Cross-provider prompt-guide reuse.** `veo_video_prompt_guide` is valuable whether the caller reaches Veo via `gemini_generate_video` or `fal_generate_video`. One guide per model, not per provider.
- **Env-key-aware discovery.** `model_catalog` reads configured API keys and returns only models the caller can actually invoke right now. No guessing, no 401s mid-pipeline.
- **No ossification.** Adding a new model is a catalog entry + a prompt guide. Adding a new provider is a new transport tool. Neither churns the rest of the surface.

## Constraints

- **Open source project already shipped.** API changes are additive. Existing `gemini_*` tools stay. Additions (transport tools, catalog, new guides) cannot break current callers.
- **MCP tool surface is LLM-readable.** Tool descriptions must carry enough signal for callers to use them without reading fal's or Google's docs out-of-band. The prompt-guide-per-model pattern carries most of that load.
- **Multi-provider is the steady state.** No forced consolidation, no hidden cross-provider routing.
- **Budget:** total validation + Phase 1 spend ≤ ~$10 across Q1 docs read, Q2 bake-off, and Phase 1 smoke. Phase 4+ budgets TBD per bake-off when those phases land.

## Terminology

- **Capability** — video, image, music, speech, safety. Top-level axis of the catalog.
- **Sub-class** — within a capability, a use-case grouping that has distinct quality criteria. Today's sub-classes: `video.talking_head`, `video.cinematic`, `image.general`, `music.general`, `speech.general`, `safety.text`. New sub-classes get added when a pipeline use case exposes a distinct criterion (e.g., `image.portrait` if a portrait bake-off reveals portrait-specific winners).
- **Model (logical)** — a logical model identity like "kling-ai-avatar-v2-pro" or "veo-3.1". The thing whose prompt language is stable across access routes. Referred to as `slug` in the catalog JSON.
- **Access route** — a specific (provider, transport tool, wire-level model id) triple through which a logical model can be invoked. A single logical model may have multiple access routes (e.g., Veo via Google or via fal).
- **Transport tool** — `fal_generate_video`, `gemini_generate_video`, etc. Thin passthrough; requires an explicit `model` param (wire-level id).
- **Non-transport tool** — `fal_classify_text`, `model_catalog`, etc. Purpose-built tools that may have deterministic defaults.

## Core Principles

The five principles that govern every decision in this design. Every subsequent section is an application of these.

1. **Transport tools require `model` explicitly.** `fal_generate_video`, `gemini_generate_video`, and future `provider_*_*` transport tools are thin passthroughs with no server-side defaults. **Non-transport tools** (safety classifiers, `model_catalog` itself, future narrow-purpose tools) may default deterministically when exactly one registered entry matches the tool's scope — but the rule is "exactly one," not "obvious choice." The moment a second entry exists, the default becomes `required`.
2. **Model selection is the caller's job, informed by the documentation layer.** Callers discover via `model_catalog`, learn via per-model prompt guides, invoke via the matching transport tool. Three-step flow, all explicit.
3. **Prompt guides are keyed by model, with per-route caveats.** The prompt *language* (what goes in the `prompt` string) is stable across access routes for the same logical model — `veo_video_prompt_guide`'s prose guidance works whether Veo is reached via Google or fal. Per-route differences exist in **request-shape** (which config knobs each provider surfaces — e.g., Google's SDK exposes `generate_audio`, `person_generation`, `seed` that fal's wrapper may not) and are documented in each guide's `Access routes` section as per-route notes.
4. **The catalog is the single source of truth for what's callable.** Static JSON in-repo, env-key filtered at read time. Every entry has a real prompt guide (or `null` for safety-class entries) and a real, registered transport tool. Entries pointing at unregistered tools are filtered out at runtime with a warning. New model = PR with catalog entry + guide + any transport updates.
5. **No routing, no taste baked into artificer.** The catalog surfaces cost and capabilities as data; the caller decides. No `generate_video(model="veo-3.1")` magic that auto-picks the cheapest access route.

## Premises (agreed 2026-04-23)

1. **Silence problem is architecturally solvable** by any audio-decoupled model. Confirmed by Q2 bake-off: Kling Avatar and veed/fabric both preserved 1s tail silence on Jenn's avatar.
2. **`extra_params: Record<string, unknown>` passthrough** is the right shape for transport tool schemas. Fal's cross-model JSON is unified; Google's SDK surface is already typed per-call. Per-model specificity lives in prompt guide tools.
3. **`@fal-ai/client` SDK is pinned.** Version bumps are explicit events with a testing checklist including re-running the relevant-capability bake-off.
4. **Multi-provider is the long-term state.** No retirement trigger for existing `gemini_*` tools. Artificer is a connection + schema + prompt-guide + discovery layer.
5. **Validation gates the build.** Per-capability bake-offs answer "which model?" before implementation commits.
6. **Use cases split within a capability.** Talking-head is not cinematic video; portrait image is not scene image. Each sub-class deserves its own bake-off when a pipeline use case emerges. The catalog structure reflects sub-classes.
7. **Cross-provider model availability is real.** Same model via multiple providers (Veo via Google or fal; Gemini image via Google or fal). The catalog surfaces all access routes for a model; callers pick explicitly.

## Scope — IN vs OUT (Commitment)

This design is a commitment to a specific scope. Items in the OUT list are not "won't do ever" — they're "won't design in this doc." Any of them becomes a separate plan file when the need is concrete.

### IN

- Core principles codified in this doc
- `fal_generate_video` transport tool (Phase 1)
- `model_catalog` tool — env-key filtered, static-JSON-backed (Phase 2)
- Prompt-guide format spec (see next section) with retrofits for existing `veo_video_prompt_guide` and `gemini_nanobanana_prompt_guide` (Phase 2)
- New prompt guides for Phase 1 talking-head class: `kling_avatar_prompt_guide`, `veed_fabric_prompt_guide` (Phase 2)
- `fal_classify_text` safety tool (Phase 3)
- `fal_generate_image` transport tool + image-class catalog entries + per-model guides (Phase 4, gated on pipeline use case)
- `fal_generate_music` + `fal_generate_speech` transport tools + `music.*` / `speech.*` catalog entries (Phase 5, gated on pipeline use case)

### OUT (parked — separate plan if needed)

- **Dynamic catalog fetching.** No scraping fal's or Google's live model lists. Static JSON only.
- **Profiles / named preferences.** No `profile: "talking_head_default"` or similar abstraction.
- **Hidden routing / auto-select-cheapest-provider.** Transport tools never magic across providers.
- **Model deprecation automation.** No stale-guide detection, no version-bump migration helpers. Manual PR process.
- **Multi-API-key rotation, quota tracking, billing aggregation.** Out of scope.
- **General-purpose evaluation harness.** Today's `scripts/fal-bakeoff.mjs` is the MVP. Generalizing into a reusable capability-agnostic harness is a separate plan, not blocked by this one.
- **Cinematic-video bake-off, image bake-off, music bake-off, TTS bake-off.** Each is a future plan tied to a pipeline use case.

Any subsequent conversation that proposes adding an item above to this doc should be redirected to a new plan file.

## Prompt Guide Format Spec

All prompt guide tools (existing and new) conform to this structure. Enables catalog indexing and consistent caller experience.

**Tool naming:** `{model_slug}_{capability}_prompt_guide`, e.g. `veo_video_prompt_guide`, `kling_avatar_prompt_guide`, `flux_image_prompt_guide`.

**Return value structure** (prose output, sections in this order):

```
# {Model Display Name} Prompt Guide

## What this model is best for
{1-3 sentences on ideal use cases}

## Known strengths
- Bullet list

## Known weaknesses / quirks
- Bullet list

## Input requirements
- Image format, audio format, URL constraints (public vs signed), size limits
- Specific requirements that must hold for the call to succeed

## Prompt structure
{Prose guidance: shot descriptors, camera directions, style keywords, what NOT to include}

## Example prompts
{2-3 concrete examples with short captions explaining the choices}

## Access routes
{Table of {provider, tool, model_id, cost, notes} — this is machine-parseable and catalog-readable}

## Last verified
{Date + artificer version + what was verified (e.g., "prompt structure from 2026-04-23 bake-off")}
```

**Retrofits for existing guides:** `veo_video_prompt_guide` and `gemini_nanobanana_prompt_guide` get updated in Phase 2 to match this format. The `Access routes` section is the new part — previously those guides assumed Google-only access.

## `model_catalog` Tool Design

Top-level artificer tool, not scoped to any provider.

**Signature:**
```typescript
model_catalog({
  capability?: 'video' | 'image' | 'music' | 'speech' | 'safety',
  include_unavailable?: boolean,  // default false — show models whose keys aren't configured
})
```

**Return shape** (JSON):

```json
{
  "video": {
    "talking_head": [
      {
        "slug": "kling-ai-avatar-v2-pro",
        "prompt_guide": "kling_avatar_prompt_guide",
        "access_routes": [
          {
            "provider": "fal",
            "tool": "fal_generate_video",
            "model": "fal-ai/kling-video/ai-avatar/v2/pro",
            "cost": "$0.115/sec",
            "available": true,
            "key_env_var": "FAL_KEY"
          }
        ]
      },
      {
        "slug": "veed-fabric-1.0",
        "prompt_guide": "veed_fabric_prompt_guide",
        "access_routes": [
          {
            "provider": "fal",
            "tool": "fal_generate_video",
            "model": "veed/fabric-1.0",
            "cost": "$0.08/sec (480p); $0.15/sec (720p)",
            "available": true,
            "key_env_var": "FAL_KEY"
          }
        ]
      }
    ],
    "cinematic": [
      {
        "slug": "veo-3.1",
        "prompt_guide": "veo_video_prompt_guide",
        "access_routes": [
          {
            "provider": "google",
            "tool": "gemini_generate_video",
            "model": "veo-3.1-generate-001",
            "cost": "per-video pricing; see Google Cloud pricing page",
            "available": true,
            "key_env_var": "GOOGLE_API_KEY"
          },
          {
            "provider": "fal",
            "tool": "fal_generate_video",
            "model": "fal-ai/veo3.1/image-to-video",
            "cost": "per-second pricing; see fal model page",
            "available": true,
            "key_env_var": "FAL_KEY"
          }
        ]
      }
    ]
  },
  "image":  { /* Google routes seeded in Phase 2 with `stub: false`; fal routes added in Phase 4 */ },
  "music":  { /* Google route for Lyria seeded in Phase 2 with `stub: false`; fal routes added in Phase 5 */ },
  "speech": { /* Google route for Gemini TTS seeded in Phase 2 with `stub: false`; fal routes added in Phase 5 */ },
  "safety": {
    "text": [
      {
        "slug": "qwen-3-guard",
        "prompt_guide": null,
        "access_routes": [
          { "provider": "fal", "tool": "fal_classify_text", "model": "fal-ai/qwen-3-guard", "cost": "$0.002 per 1K tokens", "available": true, "key_env_var": "FAL_KEY" }
        ]
      }
    ]
  }
}
```

**Field semantics:**
- `slug` — logical model identifier, stable across access routes. Callers use this to select which model they want conceptually; they pass the route-specific `model` string to the transport tool.
- `model` (per route) — the wire-level id the transport tool needs. What the caller passes to `fal_generate_video({ model: "..." })` or `gemini_generate_video({ model: "..." })`. Different string per route even for the same logical model.
- `cost` — free-form string describing pricing. No numeric field because providers price differently (per-second, per-video, per-token, per-image, tier-based). Callers who need to compare costs programmatically parse the string or defer to the prompt guide.
- `key_env_var` — required. The env var artificer checks to compute `available`.
- `available` — `true` if `key_env_var` is set AND the transport tool is registered at runtime. Entries failing either check are excluded from the default response; include them via `include_unavailable: true`.
- `stub` (optional) — `true` indicates the entry is seeded for future activation (transport tool not yet shipped). Stubbed entries are always filtered out unless `include_unavailable: true`. Prevents Phase 2's catalog from misleading callers about capabilities that land in Phase 4/5.

**Zero-keys-configured behavior:** if no keys are set at all, return the full structure with all `available: false` and an additional `warnings: ["No API keys configured. Set FAL_KEY, GOOGLE_API_KEY, etc. and retry."]` top-level field. Covered by test.

**`extra_params` collision with the structural URL params:** if a caller passes `extra_params: { image_url: "..." }` without the tool's `image` param, the value passes through cleanly (there's nothing for the structural path to collide with). If both `image` and `extra_params.image_url` are set, the resolved `image_url` from `image` wins and a collision warning logs. Same rule for `audio` / `audio_url`. Unit test covers both cases.

**Data source:** `src/catalog/models.json` committed in-repo. Read once at tool invocation, filtered by `process.env` and by transport-tool registration, returned.

**Maintenance rule:** every catalog entry must have `prompt_guide` pointing at a registered tool (or `prompt_guide: null` only if `capability === "safety"`) and every access route must point at a registered transport tool. CI test asserts this; runtime filter protects callers against drift between `models.json` and actual tool registration.

**LLM caller flow:**
```
1. model_catalog({ capability: 'video' }) → see what's available
2. Pick a logical model based on use case (talking_head vs cinematic) + cost + availability
3. {chosen model's prompt_guide}() → learn how to prompt it
4. {chosen access route's tool}({ model, prompt, ... }) → generate
```

## Recommended Approach — Approach A, sequenced

Phase 0 (already complete as of 2026-04-23) fed the design; Phases 1–5 ship the architecture.

### Phase 0 — Architecture screen + talking-head bake-off (COMPLETE)

**Q1 (docs read, ~1 hour, $0)** — captured in `docs/plans/fal-q1-research-2026-04-23.md`. Output: shortlist of 3 audio-decoupled candidates (Wan 2.7, Kling AI Avatar v2 Pro, veed/fabric-1.0); infra posture for URL handling, concurrency, TTL, uploads.

**Q2 (scripted bake-off, afternoon, ~$5 capped)** — captured in `docs/plans/fal-bakeoff-2026-04-23.md`. Output: Kling AI Avatar v2 Pro as default, veed/fabric-1.0 as fallback, Wan 2.7 deferred (root cause: `generation_timeout` on fal's Wan runner at default 1080p audio-driven mode — 2723s of inference time before fal's internal timeout; workaround is explicit `resolution: "720p"` per bake-off doc's post-session diagnosis).

### Phase 1 — `fal_generate_video` transport tool

1. Add `@fal-ai/client` to `package.json` with **exact version pin**. Add note to `CONTRIBUTING.md`: upgrading this SDK requires re-running the talking-head bake-off against the current default model.
2. Create `src/generation/fal/client.ts` — factory, reads `FAL_KEY`, configures SDK once (module-level singleton, same pattern as Veo). Missing-key error shape matches `GOOGLE_API_KEY` error shape (test covers this).
3. Create `src/generation/fal/types.ts` — Zod schemas, `extra_params: z.record(z.unknown()).optional()`.
4. Create `src/generation/fal/video.ts` — `fal_generate_video` tool:
   - Accepts: `model` (string, **required** — no env default), `prompt`, `output`, `image?`, `audio?`, `duration_seconds?`, `aspect_ratio?`, `resolution?`, `negative_prompt?`, `extra_params?`, `poll_timeout_seconds?` (default 300, matching `gemini_generate_video`).
   - **Input URL resolution** (the part the Veo flow doesn't cover): fal's API accepts `image_url` and `audio_url`. For each of `image` and `audio`:
     - If already a public HTTPS URL: pass through. (Q2 confirmed `storage.googleapis.com/...` public URLs work without upload for Kling Avatar and veed/fabric. Behavior may differ per model — document per-model in the prompt guide.)
     - Otherwise, `resolveInput()` → get local path → wrap in `Blob` → `fal.storage.upload(blob)` → pass returned URL. Covers `gs://`, `s3://`, local inputs.
   - **Fal payload construction** — explicit mapping:
     | Tool param | Fal input key |
     |------------|----------------|
     | `prompt` | `prompt` |
     | (resolved image URL) | `image_url` |
     | (resolved audio URL) | `audio_url` |
     | `duration_seconds` | `duration` |
     | `aspect_ratio` | `aspect_ratio` |
     | `resolution` | `resolution` |
     | `negative_prompt` | `negative_prompt` |
     | `extra_params.*` | spread as top-level keys |

     **Key precedence:** top-level mapped keys win over `extra_params`. Collision logs a warning. Unit-tested.
     Keys the model doesn't accept are passed through to fal; fal's 4xx response is surfaced verbatim to the caller (no silent drops).
   - Call `fal.subscribe(model, { input, logs: true })`. `poll_timeout_seconds` → SDK's native `timeout` (ms). No `Promise.race` needed per Q1 research.
   - Download returned `video.url` to `output` via `getProvider(output).write()`. MIME type from HTTP `Content-Type` response header; fall back to `video/mp4`.
   - **Fal safety/content-filter rejection** surfaces as structured error with fal's reason preserved. Error shape matches Veo's `raiMediaFilteredReasons` pattern.
   - Return `{ uri: output }`.
5. Register in `src/generation/fal/index.ts` → `src/generation/index.ts` → `src/index.ts`.
6. Unit tests (Vitest): mock `@fal-ai/client`. Cover payload mapping, `extra_params` collision precedence + warning, output write + MIME handling, missing-key error, fal 4xx, fal 5xx, timeout, safety-rejection error shape, HTTPS-passthrough vs upload-required paths.
7. Integration smoke (`INTEGRATION=1`, ~$0.60): one I2V generation with Jenn's avatar via Kling AI Avatar v2 Pro. Confirms silence preservation in-tool (not just in bake-off script). Plus a two-call parallel smoke to confirm concurrency.
8. **Deliberate-failure billing probe** (Q1 unresolved item): one submission with an invalid audio URL to see whether fal charges for dispatch-time failures. Document result.

### Phase 2 — Catalog + prompt guide retrofits + new talking-head guides

1. **Prompt guide format spec** (section above) committed as a convention doc: `docs/conventions/prompt-guides.md`.
2. **Veo-via-fal prompt-parity smoke** (~$1): before retrofitting `veo_video_prompt_guide` to list both Google and fal access routes, run a single cinematic-prompt generation via `fal-ai/veo3.1/image-to-video` and compare the output against a matched Google-Veo generation from the same prompt. Purpose: validate that the prompt *language* is stable across routes (Principle #3). If outputs differ materially, the retrofitted guide gets per-route prompt notes; if they match, publish with a single "Prompt structure" section and per-route notes only for request-shape.
3. **Retrofit existing guides** to the spec: `veo_video_prompt_guide`, `gemini_nanobanana_prompt_guide`. Add `Access routes` sections with per-route request-shape notes (e.g., "Google-route exposes `generate_audio`, `person_generation`, `seed`; fal-route does not").
4. **New talking-head guides:**
   - `kling_avatar_prompt_guide` — content derived from Q2 bake-off observations, fal's model page, Kling's official prompting docs.
   - `veed_fabric_prompt_guide` — same, plus explicit 480p vs 720p cost note.
5. **`model_catalog` tool** implementation in `src/catalog/catalog.ts`:
   - MCP tool description string: "List available media-generation models grouped by capability and sub-class. Returns logical model slugs, prompt guide tool names, and access routes (provider + transport tool + wire-level model id + cost). Call this first when you need to pick a model for a task; then call the relevant prompt_guide, then the access route's transport tool. Use `include_unavailable: true` to see models whose API keys aren't configured."
   - Reads `src/catalog/models.json`.
   - Filters by env keys (reads `FAL_KEY`, `GOOGLE_API_KEY`, per-route `key_env_var`) AND by transport-tool registration (drops entries whose `tool` isn't in the MCP server's registered tool list, logging a warning server-side).
   - Zero-keys-configured behavior per spec above (full structure with all `available: false` + `warnings` field).
   - `include_unavailable: true` returns everything including stubs.
   - Returns shape per spec above.
   - Unit tests: filter correctness per key configuration, capability param, `include_unavailable` flag, zero-keys warning, stub filtering, runtime-missing-tool filtering.
6. **Seed `models.json`** with:
   - Talking-head entries for Kling AI Avatar v2 Pro, veed/fabric-1.0 — `stub: false` (transport ships in Phase 1).
   - Cinematic entries for Veo 3.1 (Google route `stub: false`, fal route `stub: false` since `fal_generate_video` ships in Phase 1), Wan 2.7 (fal route `stub: false` with a prompt-guide note mandating `resolution: "720p"` for audio-driven mode to avoid `generation_timeout` at 1080p).
   - Image entries for Gemini 2.5 / Nano Banana (Google route `stub: false` since `gemini_generate_image` already ships). Fal image route entries are NOT seeded — they land in Phase 4 with the `fal_generate_image` transport.
   - Music entry for Lyria (Google route `stub: false` since `gemini_generate_music` already ships).
   - Speech entry for Gemini TTS (Google route `stub: false` since `gemini_generate_speech` already ships).
   - Safety entry for Qwen 3 Guard (fal route `stub: true` until Phase 3 ships `fal_classify_text`).
7. CI test (`src/catalog/catalog.test.ts`): every entry's `prompt_guide` resolves to a registered tool (or is `null` with `capability === "safety"`); every access route's `tool` resolves to a registered tool OR has `stub: true`; every route has a `key_env_var`; every route has a `cost` string. Fails the build on drift.

### Phase 3 — Safety classifier (`fal_classify_text`)

1. Create `src/generation/fal/safety.ts` → `fal_classify_text` tool.
2. Input: `text`, optional `model`, optional `categories[]`. **Default rule (non-transport tool):** if `model` is omitted AND `catalog.safety.text` contains exactly one entry, use that entry's fal-route `model` value. If the catalog has more than one safety.text entry at invocation time, `model` becomes required. Rule is deterministic and covered by test; "obvious choice" is not a factor.
3. Returns `{ safe: boolean, categories: Array<{name, confidence}>, raw: unknown }`.
4. Unit tests with fixtured harmful/safe inputs, plus tests for the one-entry auto-default AND the multi-entry-requires-model branches.
5. Integration smoke: 3 hand-picked inputs.
6. Catalog update: flip Qwen 3 Guard entry from `stub: true` to `stub: false`.
7. Pipeline-side integration into `content_vet` is out of scope (separate PR in `social-content-pipeline`).

### Phase 4 — Image generation (gated on pipeline use case)

Triggered when a concrete pipeline need emerges (blog covers, carousel backgrounds, product shots). Scope:

1. Class bake-off (portrait, landscape, product, etc. — define classes based on the triggering need).
2. `fal_generate_image` transport tool — same shape as `fal_generate_video`.
3. Prompt guides per chosen model (Flux, Seedream, Imagen-on-fal, etc.).
4. Catalog update to activate image entries.

### Phase 5 — Music + TTS (gated on pipeline use case)

Same template as Phase 4 but for `fal_generate_music` and `fal_generate_speech`. Ships when a pipeline use case specifically requires a non-Google music or TTS model.
